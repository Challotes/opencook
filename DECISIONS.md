## Identity & Security

### Current state (Phase 1 — acceptable for now)
- BSV keypair generated in-browser via `PrivateKey.fromRandom()`
- Stored as plaintext WIF in localStorage
- No encryption, no password, no PIN
- Acceptable only because no real money is at stake yet

### The hardware problem (settled)
- BSV uses secp256k1 elliptic curve
- No hardware chip supports secp256k1 directly (not Apple Secure Enclave, not TPMs, not passkeys, not YubiKeys)
- Hardware can't **be** the BSV key, but hardware can **guard** it via encryption wrapping

### Planned upgrade path (6 stages, settled)
1. **Fix Now:** Server-side signature verification, rate limiting, try/catch on JSON.parse, hide WIF from DOM, CSP headers
2. **Stage 1 (current):** Raw localStorage with renamed labels
3. **Stage 2:** Passphrase encryption — download backup first, then set passphrase, AES-256 encrypt localStorage. Fresh key at upgrade time
4. **Stage 3:** Passkey wrapping — WebAuthn PRF replaces passphrase. Firefox falls back to passphrase. HKDF domain separation
5. **Stage 4:** Self-funded posting — UTXO check, client-side tx building, server fallback
6. **Stage 5:** Revenue + daily limits — 5 free posts/day, QR to fund, fairness agent routes revenue
7. **Stage 6:** Server HSM / threshold signing — required before significant funds flow

> **[PARTIALLY SUPERSEDED 2026-06-13 — see "Key rotation REMOVED in favor of encrypt-in-place" later in this file.]** The "5-minute window problem" and "Key rotation on upgrade" subsections below describe the ROTATION-based design (`MoveAddressModal`, a NEW key on upgrade, an on-chain migration chain) — all REMOVED. **Current model:** the key/address NEVER changes; adding a passphrase encrypts the EXISTING key in place (`encryptInPlace`) — no rotation, no migration, no `MoveAddressModal`. The plaintext-window risk is now mitigated by gating recovery-file export behind setting a passphrase (no unencrypted file ever leaves the device), not by rotation. (Passkey-wrapping, further below, REMAINS a valid future enhancement — it wraps the same key.) These subsections are retained as historical record of the original design.

### The 5-minute window problem (settled + implemented)
- Any key that existed as plaintext in localStorage must be assumed potentially compromised
- **Implemented:** Protection happens via `MoveAddressModal` (the "Passphrase" row of the You modal, also the entry point from the "Not protected" red banner). The wizard generates a NEW key, the old key signs a migration, and the new key is stored AES-256-encrypted.
- Migration posted on-chain via OP_RETURN — permanent verifiable link from old pubkey to new pubkey
- All contribution history and future payments follow the migration chain
- Old key becomes useless after migration — someone stealing it gains nothing

### Security upgrade model (settled)
- **Self-service:** Unprotected users see a red "Not protected" banner at the top of the identity dropdown that opens `MoveAddressModal` directly. Protected users use the same wizard via the "Passphrase" row in the You modal.
- **Deferred activation (future):** System nudges users when earnings reach a threshold (e.g., $5) but haven't upgraded yet
- **Zero friction for new users:** No passphrase required on first visit. Upgrade is optional and user-initiated
- **Key rotation on upgrade:** Fresh keypair born encrypted, never existed as plaintext

### Passkey-wrapped keys (chosen approach)
- BSV key encrypted with AES-256
- Decryption key derived from WebAuthn PRF extension tied to biometrics
- Stolen localStorage = useless ciphertext
- Works on Chrome, Safari, Edge; Firefox needs passphrase fallback
- Medium implementation effort, best security/UX tradeoff

## Self-Funded Posting (settled)

- Server pays for posts by default (~0.00001 BSV per post)
- When user has BSV balance, app silently switches to user-funded
- Same button, same UX — funding source switches invisibly
- Must create change output or user loses remaining balance
- Cost: ~1 satoshi per post; 10,000 satoshis covers thousands of posts

## Anti-Spam (settled direction)

- Free posts capped per day (5/day suggested)
- Under limit: server pays, no friction
- Over limit with balance: self-funded, no friction
- Over limit without balance: "You've got more to say" + QR code
- Server-side enforcement (pubkey + IP + session token), not chain-only
- Optional: proof-of-work for free posts

## Bootboard (settled)

- **Mechanic:** Any post can be "booted" to a spotlight slot by paying a fee. Someone else pays, you get booted off
- **Boot count:** Tracked per post — shows how many times a post has been featured
- **Revenue model:** Built into the UX, not bolted on. Creates natural urgency and competition
- **Animations:** Shake + glow + slide-in on holder change. Expandable history
- **Boot icon:** Uses 🥾 emoji (custom SVG attempted, reverted to emoji for clarity at small sizes)

## Agent Chat (settled)

- **AI-powered:** Uses Claude Haiku 4.5 via Anthropic API with streaming SSE responses
- **System prompt:** Single source of truth in `src/data/agent-prompt.ts`
- **Endpoint:** `/api/agent` route handler (POST, streams text chunks)
- **Cost:** ~$0.001 per question (~25,000 questions per $25 credits)
- **Rate limiting:** 30 requests/min per IP + max 3 concurrent requests (prevents Anthropic API overload)
- **Input limits:** Max 20 messages, 2000 chars each per request
- **Location:** "Ask AI" pill button below compose box, opens as centered modal (bottom sheet on mobile)
- **Post button:** Telegram-style — mic icon when empty, amber send arrow when text is present

## Genesis Section (settled)

- **Founding conversation** preserved at top of feed as immutable record
- **Visited state** persisted in localStorage — shows full "Genesis" pill first visit, discreet chevron after
- **Fairness agent tie-in:** This is the starting point for contribution tracking
- **NOT collapsible — by design.** Genesis is feed content, not a UI widget. It lives at the top of the scroll area. Users discover it by scrolling up (via the Genesis button), not by toggling a panel. Do not add a collapse/expand toggle.

## Feed UX Model (settled)

- **Telegram-style:** User enters at the most recent post (bottom of feed). Feed grows upward.
- **Unread tracking:** When user leaves and returns, new posts accumulate. Unread counter badge shows on the scroll-to-bottom button. IntersectionObserver marks posts as read when they scroll into view.
- **Navigation:** Scroll-to-bottom button (with unread count) and genesis chevron (scroll to top) are the two navigation anchors. Users explore the full history by scrolling between them.
- **Mobile enter-to-post:** The Telegram-style mic→arrow toggle on the compose button is the primary affordance. The "Enter to post" text hint is desktop-only — this is intentional, not a bug. Mobile users tap the amber arrow.
- **No collapse, no accordion, no "read more" gates** on any feed content. The feed is a continuous scroll.

## Agentic Fairness (settled direction)

- **Phase 1:** Human-defined parameters, AI executes (current target)
- **Phase 2:** AI suggests parameter changes, humans approve
- **Phase 3:** AI adjusts within bounds, humans can override
- **Phase 4:** Fully agentic, humans only intervene on disputes
- The name describes the vision, not just today's implementation
- **Revenue model:** Boot fees split directly to contributors via multi-output BSV transaction. See **FAIRNESS.md** for the full model, formula, parameters, and gaming analysis
- **This is a demo model** — simple post-count + engagement + recency. Will evolve as real value contributions emerge (code, design, community). The point is proving the mechanism works first

### Revenue Distribution Rules (settled)
- **True no-custody:** Every sat in = every sat out in the same transaction. No database balances, no pending payouts, no IOUs. Even 1-sat shares get a UTXO output.
- **Boots require signed identity:** Booter must have a pubkey. Prevents free boot abuse (can't fake a new identity to get more free boots). Unsigned users can post but not boot.
- **Only signed posts are boostable:** The creator bonus (15%) needs an address to pay. Unsigned posts can't be booted. Encourages identity adoption.
- **Zero pool recipients → creator gets 95%:** If nobody qualifies for the pool (e.g., only 1 contributor who is the creator), the 80% pool goes to the creator. 5% still goes to platform.
- **Dynamic pricing formula:** `boot_fee = max(1000, min(250000, active_contributors × 156))`. Active = posted in last 30 days, counted by pubkey only. Price cached 1 hour. Rationale: 156 ensures bottom-25% contributors clear meaningful payouts under Pareto distribution.
- **Free boots:** First 15 per pubkey, tracked in SQLite `boot_grants` table. Server wallet pays at the **floor price** (1,000 sats), not the dynamic price. After 15, user funds their address via QR code. See "Free boots pay floor only" decision below for rationale.
- **$50/month operator budget:** Covers subsidised boots, on-chain posting, hosting, API. Sustainable through ~200 users, then user-paid boots and 5% platform cut take over.
- **Trustless P2P payments:** Paid boots are built client-side — user's browser constructs the multi-output BSV transaction directly to every contributor. Server never touches user funds. Server only provides the contributor list. The transaction itself is the verifiable proof. Free boots use the server wallet (server is the payer, not custodian).
- **Auto-switch:** Free boots → server pays. User has BSV balance → client builds trustless tx. No balance → show fund address QR. One click, ~800ms, zero custody.
- **Currency display:** Default is dollars ("Noob Mode"), toggle to sats ("Goat Mode"). Persisted in localStorage. Applies to chip balance, activity feed, and boot prices. BSV price from WhatsOnChain API, cached.
- **Identity import cleans up migrations (~~settled~~ SUPERSEDED 2026-05-26 by E29).** Original rule: "when importing an old key, any migration records pointing away from that key are automatically deleted." This is NO LONGER the policy. Old keys with forward migrations are now BLOCKED from restore entirely — `cleanupMigrations` is no longer called by any UI path. See entry below: "Restore of rotated keys is blocked outright (E29, Design C-strict)."
- **Security upgrade forces backup download (settled).** Auto-downloads the new key backup BEFORE the upgrade is considered complete. User cannot lose access to upgraded key.
- **Free boots are one-time only (settled).** 15 free boots per identity, never reset. Once used, user pays from their earnings or funded balance. No monthly reset, no balance-based gating. The 15 boots are an onboarding gift — "try before you buy." After that, the economy takes over.
- **Free boots pay floor only (settled 2026-04-09).** The server wallet pays the boot price floor (1,000 sats) on free boots, regardless of the current dynamic price. Supersedes the prior "dynamic price" rule. **Rationale:** per-user subsidy cost is now bounded at ~15,690 sats (15 × ~1,046) forever, independent of platform scale or contributor count. At BSV $25 that is ~$0.004/user; at BSV $100, ~$0.016/user — safely within the $50/month operator budget across all realistic price ranges. Dust concerns are handled by the existing split model (`split.ts:49` — every non-zero share is paid, down to 1 sat), by the sqrt × decay weight curve which naturally concentrates value on top contributors, and by the fact that free boots are a symbolic onboarding gift, not a full economic event. The real economy happens on paid boots where dynamic pricing (`contributors × 156`) keeps pool shares at ~125 sats each. **Trade-off:** contributors earn less per free boot than per paid boot. This is intentional — it preserves the onboarding gift framing without scaling the server subsidy cost with platform success. **Alternatives rejected:** tapering free-boot count (violates the "one-time only" decision above, has Sybil attack surface via contributor count inflation, creates UX unfairness between launch-day and later users, and breaks Phase 2 agent governance by making `freeBootsPerUser` non-constant), batching sub-dust payouts (breaks the trustless no-custody model), and top-K concentration (unnecessary — the sqrt curve already concentrates naturally). A GorillaPool miner fee deal is pursued separately as an optional optimization, not a dependency.

- **Live boot counts (settled 2026-06-23, Phase 8 QA #4).** The 5s `/api/posts` poll carries an additive `counts` param = the ids of CONFIRMED visible posts (capped 100); the server returns a lightweight `{id, boot_count}[]` (`getPostCounts`, only id+count — NOT full post bodies) folded into the existing `Promise.all`. The client merges these into the same atomic `setPosts` as the `tx_id` patch. **Why:** a post permanently leaves the `pending_tx` refresh set once it gains a `tx_id`, so its boot_count froze at load time — a boot from the Bootboard / another user / a server-funded boost never updated the display until reload. This makes counts authoritative-live from ALL sources. **Reconciliation:** `BootButton` resets its optimistic `+1` when the incoming `bootCount` prop advances (the authoritative count already includes the user's own boot), preventing a one-tick double-count. Purely additive — the `pending_tx`/`since_id` contract is unchanged, the `sweepOrphans()` ambient trigger is untouched, and `/api/posts` stays unrate-limited by design. Cost is trivial (~3KB/poll, a PK-indexed `WHERE id IN (...)`). Superset fix (also resolves the Bootboard-reboot-count case).

- **Mic: record + Groq Whisper, NOT the Web Speech API (settled 2026-06-25, Phase 8).** The voice-to-text mic was rebuilt. The old `webkitSpeechRecognition` (Web Speech API) is UNFIXABLE on iPhone — WebKit deferred it in home-screen PWAs indefinitely (bug #225298, "resolved later"), it's blocked in all non-Safari iOS browsers (WKWebView), absent in Firefox, and depends on iOS system Dictation being on (the `service-not-allowed` we hit). No code change fixes it — it's Apple policy. ChatGPT's web mic is confirmed (OpenAI help docs) to use server-side Whisper, not the browser API. **New architecture (the "ChatGPT way") — works on iPhone Safari + installed PWA + Android + desktop + Firefox:** `src/hooks/useVoiceToText.ts` records audio (`getUserMedia` + `MediaRecorder`) and POSTs it to `src/app/api/transcribe/route.ts`, which forwards to **Groq Whisper Large v3 Turbo** (OpenAI-compatible endpoint) and returns `{ text }`; `PostForm` inserts it via the kept `dispatchEvent('input')` trick (the one piece that makes a programmatic `.value` write visible to React). **iOS must-dos baked in:** `getUserMedia` called synchronously in the tap handler (user-gesture requirement); runtime MIME detection (`MediaRecorder.isTypeSupported` — iOS only supports `audio/mp4`); **`recorder.start(1000)`** (1-second timeslice — a single unchunked Safari mp4 blob otherwise garbles Whisper); drop empty chunks; filename extension matched to the codec. **Cost:** Groq free tier 2,000/day (no card) covers launch, ~$0.04/hr of audio after — negligible. The route mirrors `/api/agent`'s cost guards (per-IP rate limit + concurrency cap + `TRANSCRIBE_DAILY_LIMIT` circuit-breaker); returns 503 if `GROQ_API_KEY` is unset (the mic degrades to a "voice input offline" toast, nothing else breaks). **DO NOT revive the Web Speech API** — it cannot work on iPhone PWAs. **Confirmed working on ANDROID 2026-06-26; iPhone (Safari + PWA) + desktop still need testing.** Env: `GROQ_API_KEY` (+ optional `TRANSCRIBE_DAILY_LIMIT`). **Mic UX (2026-06-26):** idle mic is amber-tinted + amber ring (a discoverable voice affordance); recording is red `animate-pulse` (unambiguous "live"); transcribing is amber + spinner — one coherent control. The mic is **HANDS-FREE — it does NOT auto-open the keyboard after dictation** (text appears, send button ready); `handleTranscript` calls `el.focus()` ONLY on a fine pointer (desktop keep-typing), never on touch — so dictation never pops the keyboard or triggers the #6 scroll-off. **Post-send refocus is likewise touch-gated (2026-06-26):** `performSubmit` + the post-completion `isPending` effect refocus the textarea only on a fine pointer, so posting on touch doesn't re-pop the keyboard either (the whole compose area is hands-free on touch; desktop keeps the caret for continuous posting). Tapping the mic while a keyboard is open is ALLOWED to close it (switching to hands-free), do NOT `preventDefault` the focus-steal. **Gotcha that cost ~1h:** a persistent Groq `401 Invalid API Key` turned out to be an **UNSAVED `.env.local`** — the editor buffer showed the new key but the file on disk still held the old one, so the app read the stale key. The temp key-metadata debug log (length/prefix/suffix, never the secret) caught it: the app loaded a 68-char key while the editor showed the correct 56-char one → the on-disk value was stale. A `.trim()` on the key read is kept (whitespace defense). Lesson: if a Next env var seems wrong despite `.env.local`, confirm the file is actually **SAVED to disk** AND restart (`.env.local` is read only at server boot).

## UTXO Management (settled)

- **Fee model: 100 sat/kB everywhere (settled 2026-04-13; rate re-confirmed live 2026-06-19).** Always pass `new SatoshisPerKilobyte(100)` explicitly to `tx.fee()` across ALL tx paths — boots, consolidation, sweeps, transfers. No split fee tiers. **100 sat/kB is the current live ARC miner floor** (GorillaPool `/v1/policy` `miningFee` = 100 sat / 1000 bytes, verified 2026-06-19) — the minimum that reliably clears, NOT an overpay, so there is no safe room below it. The old split tier let *consolidation/sweeps only* run at 10 sat/kb to save ~120 sats per tx; that contributed to slow confirmations which cascaded into ORPHAN_MEMPOOL, mempool-conflict, and multi-day unconfirmed chains. **Correction (2026-06-19): the server post-logging and boot-payment paths were NEVER on the 10 sat/kb tier — they always used 100 sat/kB — so that cascade rationale never applied to them; this entry only unified the consolidation/sweep tier *up* to 100.** Verified per-post cost at 100 sat/kB (byte-exact from the @bsv/sdk fee model): ~35 sats unsigned-short / ~66 sats typical-signed / ~156 sats max-1000-char — at any realistic volume a rounding error vs the ~$50/mo budget (~217k posts/day funded at $11.62/BSV, 2026-06-19). Not worth the complexity or risk of going lower.
- **Consolidation strategy (Option F — "sweep on boot"):** `selectUtxos()` uses smallest-first ordering and includes up to 20 inputs per boot transaction. Users with many tiny UTXOs consolidate ~20 per boot for free via the change output. 290 UTXOs → fully consolidated in ~15 boots. Cost per consolidation: marginal extra tx bytes, no separate consolidation transaction needed.
- **No server custody:** Options A/C/D (server consolidation, held payouts, threshold batching) were all rejected — they require server custody or break the trustless model.
- **Dust threshold:** During UTXO selection, once boot + fee is covered, additional inputs are only included if their value exceeds the marginal fee cost of adding one more input (~15 sats at 100 sat/kb). Absolute dust is never swept.
- **MAX_CONSOLIDATION_INPUTS = 20:** 20 P2PKH inputs ≈ 2,960 bytes ≈ 296 sats at 100 sat/kb. Stays well under the boot price floor (1,000 sats minimum), leaving room for the payout in the same transaction. Tunable constant in `client-boot.ts`.
- **DUST_THRESHOLD = 16 sats (updated 2026-04-13).** Minimum UTXO value worth sweeping during consolidation. At 100 sat/kb, each P2PKH input costs ~15 sats. Was 2 sats when consolidation ran at 10 sat/kb; was 10 before that (which trapped 3–9 sat UTXOs). MAX_CONSOLIDATION_SWEEP = 200 caps pathological inputs. **Linked to fee rate:** if the fee model changes (miner partnership, rate reduction, or increase), recalculate DUST_THRESHOLD as `ceil(148 × rate_per_byte) + 1`. At 50 sat/kb → 8 sats, at 10 sat/kb → 2 sats, at 1 sat/kb → 1 sat. Also update the boot-time dust threshold in UTXO selection (~15 sats at current 100 sat/kb). Both thresholds are in `client-boot.ts`.
- **Optimistic UTXO blacklisting — REMOVED (2026-04-13).** Originally added 2026-04-10 to prevent state poisoning from failed broadcasts. Removed because it caused permanent wallet lockout: failed broadcasts left inputs blacklisted in localStorage with no automatic recovery. Double-spend prevention is now handled by: (1) mutex serializing boot calls, (2) 0-conf chaining via `_pendingChange` (own change outputs with sourceTransaction attached), (3) 3s UI boot throttle. Inputs are now blacklisted ONLY on successful broadcast.
- **3-second UI boot throttle (settled 2026-04-11).** Boot button is disabled for 3 seconds after each click. Eliminates the entire class of "user clicks faster than the network propagates" edge cases (orphan races, mempool conflicts, double-spend attempts) at zero code complexity. Architecture review recommendation — chosen over a 50-line DOUBLE_SPEND_ATTEMPTED handler that would have solved the same problem with much more complexity.
- **Reset Wallet via key rotation (settled 2026-04-11, updated 2026-04-13).** When a user wants to rotate to a fresh address, `sweepFunds` (formerly `sweepConfirmedFunds`) transfers ALL UTXOs (confirmed + unconfirmed) from the old address to the new one. At 100 sat/kb all txs confirm in next block, so unconfirmed UTXOs are safe to include. Migration is registered on-chain via existing migration.ts pipeline. Implemented as `MoveAddressModal` — a 4-stage wizard that auto-advances with visible status at each step. Label is "Move to a new address" (not "Reset Wallet") per UX review.
- **Deferred localStorage commit for key rotation (settled 2026-04-12).** `resetIdentity()` accepts `{ deferCommit: true }`. The new key is NOT written to localStorage until the caller (MoveAddressModal) confirms all stages succeeded — sweep broadcast + migration recording. Without this, a failed sweep would strand funds: localStorage updates to the new (empty) key while the old key's funds are unreachable. This bug caused 45,558 sats to be stranded during testing — recovered only because the auto-download backup of the old key (Stage 1) preserved access to the old address. This validates the design of downloading the old key BEFORE rotation as a critical safety net, not just a convenience.
- **Broadcaster: ARC everywhere (settled 2026-04-13).** All broadcast paths use ARC (@bsv/sdk default `tx.broadcast()`) — `clientSideBoot`, `consolidateUtxos`, `sweepFunds`, `autoTransferFunds`, and server wallet's `buildAndBroadcast`. Previously a mix of ARC and WhatsOnChainBroadcaster — the WoC switch was based on a misdiagnosed ARC outage (actually a local DNS cache issue, resolved by rebooting the PC). ARC sends txs directly to GorillaPool (the miner), provides structured error responses (ORPHAN status, competingTxs for double-spend), and supports 0-conf chaining via BEEF. WoC is used only for read operations (UTXO fetching, source tx hex, balance, exchange rate) — not broadcasting.
- **Server-side source tx cache (settled 2026-04-13).** `/api/tx-hex/route.ts` maintains an in-memory Map (~2000 entries, LRU eviction) of fetched source tx hex. Source tx hex is immutable, so cache-forever is correct. Eliminates repeated WoC calls for the same txid across boots, sweeps, and consolidations. Without this, a boot with 15+ inputs fires 15+ parallel WoC calls, exceeding WoC's ~3 req/s limit and causing 429 errors.
- **Explicit backup acknowledgement (settled 2026-04-16).** `backedUp` no longer flips on download trigger alone — the user must click a "Got it" confirmation after the file is offered. Applies to both paths: the You dropdown (green confirmation banner replaces the orange save-CTA) and `MoveAddressModal` (new `saved-confirm` stage gates the auto-advance to the irreversible sweep). Rationale: silent download failures (popup blocker, disk full, CSP deny, user cancels save dialog) previously masqueraded as success and cleared the warning dot — leaving users with no backup and no visible prompt. Commit `e7ecf9f`.
- **Unconfirmed-parent 404 retry in `/api/tx-hex` (settled 2026-04-16).** When WoC returns 404 on a `/tx/{txid}/hex` lookup, the parent may be a just-broadcast 0-conf chain ancestor that WoC hasn't indexed yet (typical lag: 2–10s). The proxy now retries 404s up to 3 times with 2s backoff (~6s total budget) before giving up. Kept as a separate counter from the 429/5xx retry budget so a genuinely missing tx still fails fast after the 6s window. Motivated by `fetchSourceTxsBatched` sweeps that 404'd on their own just-broadcast change parents during `MoveAddressModal` / `clientSideBoot`. Future optimization (deferred): cache own broadcast `rawTx` client-side and pass through so sweep paths never need to round-trip through WoC for their own parents.
- **Batched source tx fetches in clientSideBoot (settled 2026-04-13).** `clientSideBoot` fetches source transactions in batches of 5 with 1s inter-batch delay (matching `consolidateUtxos` pattern). Previously used bare `Promise.all` which fired all fetches simultaneously, triggering WoC rate limits on users with many UTXOs.
- **Deferred session cache in upgradeIdentity (settled 2026-04-12).** `upgradeIdentity()` no longer sets `_sessionIdentity`/`_cachedWif`/`_cachedPrivateKey` eagerly. `commitUpgrade(encStore, identity)` now accepts an optional identity and commits the session caches atomically with the localStorage write — only after `migrateIdentity()` server action succeeds. Matches the deferred commit pattern in `resetIdentity()`. Prevents in-memory signing key from diverging from server's identity record on migration failure.
- **Balance poll interval: 30s (updated 2026-04-13).** `IdentityBar.fetchLiveBalance` polls every 30 seconds (was 15s). Halves the background WoC request rate to reduce 429 pressure from normal page sitting. As of 2026-04-14 it polls `/api/balance` (server-proxied), not WoC directly.
- **Server-side WoC read proxies (settled 2026-04-14).** `/api/balance` (10s TTL, 120 req/min per IP) and `/api/unspent` (3s TTL, 180 req/min per IP) join `/api/tx-hex` as cached server proxies for all WhatsOnChain read traffic. Direct browser→WoC reads are gone. Rationale: WoC rate limits at ~3 req/s per IP; a paid boot with 15 UTXOs + background balance polls + other tabs easily exceeded this, producing 429 cascades that broke boots and froze balances. Server-side caching collapses N client requests to 1 upstream request within the TTL window and isolates the browser from WoC's rate policy. Both proxies retry 429/5xx with stale-cache fallback. IdentityBar falls back to last-known balance (not 0) on transient errors so the UI doesn't flash empty.
- **Boot confirmation via local tx parsing (settled 2026-04-14).** `/api/boot-confirm` no longer fetches the tx from WoC to verify outputs. Client sends `rawTx` alongside `txid`; server verifies `hash(rawTx) === txid` (self-authenticating) and parses P2PKH outputs locally from the raw bytes to check the split matches. Server also re-broadcasts via ARC as a safety net. Eliminates the 5–30s WoC indexing lag that previously produced false TX_NOT_FOUND errors on freshly broadcast boots, and removes another WoC rate-limit chokepoint from the critical boot path. Returns explicit `TX_CONFLICT` vs `ARC_UNAVAILABLE` error codes so the client can distinguish fatal conflicts from retriable upstream issues.
- **Structured error codes in broadcast detection (settled 2026-04-14).** Broadcast error classification matches against a structured `code` field on ARC responses, not substring search on the message. Prior substring matching against e.g. "257" produced false positives inside unrelated txids/timestamps, mislabelling successful broadcasts as conflicts. See `client-boot.ts`.
- **Server wallet shares the client's resilience stack (settled 2026-04-15, build deferred).** Server-side tx paths (`wallet.ts`, `onchain.ts`, `boot-orchestrator.ts`) must go through the same `/api/broadcast` proxy and shared WoC read cache as the browser — not direct `tx.broadcast()` or raw `fetch` to WoC. Rationale: the mutex in `wallet.ts` serializes ALL server-side on-chain activity (posts + free boots) through one broadcast call. An ARC hang without timeout freezes the entire platform for the hang duration. **The timeout must cover every in-mutex network call, not only the broadcast** — `wallet.ts` makes four un-timed external calls inside the held mutex (unspent `:89`, source-tx-hex `:167`, broadcast `:267`, double-spend recovery `:323`); any one hanging freezes posts + free boots site-wide. The same ~10s `AbortController` discipline applies to all four; on timeout, release the lock and let existing retry logic re-run. A dropped 0-conf chain fan-outs into uncached WoC reads that can trigger 429s and stall recovery. The browser was hardened against both during the 2026-04-13/14 work; the server path was not. Rule: if a resilience mitigation exists for a client path, it must also cover the server path that does the equivalent external call. See ROADMAP Phase 6.5 for build-spec (proxy, shared cache module, broadcast timeout, queue-depth metric, low-balance alert).
- **Real-time updates: SSE is enhancement, polling is ground truth (settled 2026-04-15, build deferred).** When implemented, the planned `/api/events` SSE channel and optimistic own-action updates are a UX accelerator on top of the existing 30s polling, NOT a replacement for it. Polling (`/api/balance`, `/api/earnings`, `/api/earnings?summary=1`) stays as the authoritative source. SSE events trigger an early refetch; they do not directly mutate canonical state. Rationale: SSE can drop silently (connection, edge cold-start, in-process emitter on a non-subscribed instance); optimistic balance can diverge from chain reality if a tx is orphaned or replaced. Polling self-heals both within one tick. Corollary: **do not remove the polling loops when SSE lands** — they are the fallback that makes SSE safe to fail. Also: emit-site must be wrapped in a `publishPayout()` helper so the in-process EventEmitter can be swapped for Redis pub/sub or Postgres LISTEN/NOTIFY when the app goes horizontally scaled, without touching any caller. See ROADMAP Phase 6.5 for the build-spec. **Horizontal-scale checklist (more than the emitter):** the server wallet's mutex + UTXO reservation (`_reserved`/`_pendingChange`) + double-spend blacklist (`_spent`) are also in-process memory (`wallet.ts:14,65-67`). Running 2+ instances against one `BSV_SERVER_WIF` lets instances spend the same UTXOs (double-spend errors, failed boots/posts) — a correctness break, not just a dropped UX event. Single instance is safe; multi-instance is a hard prerequisite, not a launch blocker. Cheap first step is a single global advisory lock around `buildAndBroadcast`; the full fix relocates reservation + blacklist state to shared storage. Same rule as the emitter: the swap must be possible without touching callers.
- **UTXO outpoint dedup on sweep paths (settled 2026-06-03).** `autoTransferFunds` and `sweepFunds` in `identity.ts` MUST run the raw `/api/unspent` response through `dedupeUtxos()` before building the tx. WhatsOnChain has been observed returning the same `(tx_hash, tx_pos)` outpoint twice; building a tx from that list produces `bad-txns-inputs-duplicate` peer rejection (structural malformation, not double-spend). Same pattern `client-boot.ts` already enforces via its `utxoKey` Set. Two distinct UTXOs cannot share an outpoint (Bitcoin protocol invariant), so dedup only removes structural duplicates and is safe to call unconditionally. Do NOT remove the dedup helper as "redundant" — the upstream indexer is not a contract we control, and the cost is one Map allocation per sweep. Confirmed Android-side bug 2026-06-03 with same txid `8fc71ef6…` rejected twice in a row, third attempt succeeded after WoC indexer stabilized.

Reason: Android Chrome's `100vh` includes the collapsible address bar, so `80vh` can exceed the visible area when the toolbar is showing — the user sees the modal's bottom half while the top is scrolled out of view. `svh` excludes browser chrome, ensuring the modal always fits the visible viewport. Canonical pattern: `pt-[6svh]` outer + `max-h-[80svh]` inner card. Applied site-wide 2026-06-03 across 7 centered modals: `MoveAddressModal`, `ChangePassphraseModal`, `StaleKeyModal`, `SignInModal`, `RestoreModal`, `FundAddress`, plus the `IdentityBar` You modal. The `IdentityBar` dropdown (line 1127) is `absolute`-positioned anchored under the chip — different shape, intentionally still on `max-h-[85vh]`. Static HTML recovery files (`backup-template.ts`) use **NO viewport units at all** — **CORRECTED TWICE (2026-06-26):** the original "keep 100vh, no mobile chrome problem when opened locally" assumption was wrong (Android scroll-jitter / jump-to-top as the URL bar collapses); the first fix (`100vh`→`svh` fallback) *also* didn't reliably hold — the owner's file viewer can apply the `100vh` fallback, and `svh` support in Android WebView/preview surfaces is unreliable. **Final fix:** dropped the body `min-height` entirely and moved the dark background to `html` (so a short page still shows no white area) → there is NO `vh`/`svh`/`dvh` anywhere for the collapsing Android URL bar to reflow against. The min-height was only ever doing background-coverage on short content (no vertical centering — the flex column has no `justify-content`), so removing it is layout-safe. **Do NOT reintroduce a viewport-unit min-height in the recovery file.** Browser support: Chrome ≥108, Safari ≥15.4, Firefox ≥101 — all covered. **Do NOT add new centered modals on `vh`** — anchor on this pattern.

- **Every key rotation produces an encrypted key (settled 2026-04-17).** `resetIdentity()` (plaintext key rotation) removed from primary UI. Both "Not protected" and "Move to new address" now route through `upgradeIdentity()` via MoveAddressModal, which collects a passphrase first. Users are always protected after rotation. Plaintext `resetIdentity` remains in the codebase as an emergency recovery escape hatch only. Rationale: each rotation risks chain breaks — fewer rotations (one instead of two for move-then-protect) = fewer opportunities for orphaned posts. The combined flow also downloads an encrypted backup automatically, reducing the number of recovery files a user must manage.
- **Identity sweep functions use the /api/unspent proxy (settled 2026-04-17).** `sweepFunds()` and `autoTransferFunds()` in `identity.ts` no longer hit WoC directly. They use `/api/unspent?fresh=1` which provides retry, stale-cache fallback, and rate-limit protection. Rationale: a single WoC 429 or empty response during a sweep previously caused silent fund loss — the function returned "no funds, not an error" and the rotation committed, stranding sats on the old address with the key deleted from localStorage.

If orphaned pubkeys are detected, the user is warned with a "proceed anyway" escape hatch. Rationale: migration chain breaks silently orphan posts from the fairness system, dropping earnings from ~590 to ~11 sats per split with no visible error.

Rationale: OpenCook has no email/phone recovery channel; the hint is the only lifeline if a user forgets their passphrase.

Root cause: iOS Quick Look (the WebKit-based previewer used by Files, Mail preview, Messages preview, AirDrop preview, AND macOS Finder Quick Look) does NOT execute inline JavaScript in local HTML files for security reasons. The previous template populated every dynamic field via `document.getElementById(...).textContent = BACKUP_DATA.X`, which left the file looking blank in any non-JS viewer. **Fix:** every static-renderable field is now interpolated into HTML at template-build time — name, address, saved date, plaintext WIF, hint, and footer stamp. Only the encrypted decrypt flow remains JS-dependent (it must run `crypto.subtle.decrypt`), and encrypted files now show a `<noscript>` banner instructing users to open in Safari/Chrome/Firefox if they're in a Quick Look preview. **Specifics:** (a) `formatSavedDate` helper uses fixed `en-US` locale (not `undefined`) so output is stable across server locales (Vercel, Railway, Windows dev). (b) `escapeHtml` applied to all user-controlled fields (`name`, `hint`); architect confirmed sufficient for body context — threat model is self-XSS only, not meaningful. (c) `.meta-value` gets `user-select: all` so iOS users can long-press-copy the address even when the JS Copy button is inert. (d) Dead JS removed (don't leave `meta-name`/`meta-address`/`meta-date`/`footer-stamp`/`wif-display`/`hint-text` setters that look load-bearing to future readers). (e) BACKUP_DATA JSON kept as-is in the script tag — slimming it adds risk without benefit; only the encrypted `wif_encrypted` / `oldWif_encrypted` / `oldAddress` fields are read by JS now, but harmless to leave the rest. (f) `<noscript>` banner uses amber `.noscript-banner` style (informational, not red-warning), placed above the decrypt card so users see what they're missing without the card being replaced. **Threat model unchanged.** The WIF was always in the rendered DOM after JS ran AND in the JSON inside `<script>`. Moving it to HTML body adds one more place inside the same file where it lives — but anyone with the file already has the WIF regardless. The plaintext red banner (*"This file is not encrypted. Anyone who can open it can take your account."*) renders statically in both old and new templates, so iOS Quick Look users see the warning before the WIF. Architect explicitly validated this with no security regressions. **Anti-patterns to avoid:** don't move the encrypted ciphertext into HTML body (it'd be silly — JS needs it from JSON for decryption), don't drop the `<noscript>` banner (encrypted files need clear guidance for non-JS viewers), don't switch to `toLocaleDateString(undefined, ...)` at build time (server locale is unstable).

## Platform & Distribution

- **PWA over App Store, web-first only (settled 2026-05-10).** No iOS App Store or Google Play submission. Apple's 30% IAP cut is incompatible with sat-level boot fees, Apple has historically rejected BSV/crypto-payments apps, and UGC-with-payments triggers Apple's highest moderation tier. Months of submission with low success probability is a category cost we can't absorb. Google Play alone creates an Android-only third storage sandbox that worsens identity confusion. The PWA install flow IS the "app experience" on every platform. **Anti-patterns:** retrofitting boot through Apple IAP (destroys economics), Android-native first then iOS later (creates sandbox asymmetry), Capacitor / native-shell wrappers (still triggers IAP review). **Revisit only when:** all three are simultaneously true — major traction (>5k WAU), iOS PWA notifications still painful enough to lose users, AND a clear App Store policy path opens. Until then treat as settled. Full rationale in LAUNCH_PLAN.md D1.

- **In-app browsers blocked at the door, not supported (settled 2026-05-10).** Detect known in-app browsers (Facebook, Instagram, X, TikTok, LinkedIn, WeChat, Line, Discord, Snapchat, Pinterest, Reddit, Slack, Telegram, KakaoTalk) at the root layout and serve a splash with logo + headline *"Open OpenCook in your browser"* + body *"You're inside [App]'s built-in browser. OpenCook uses your browser's secure storage. Open in your browser instead — your account stays with you."* **Hard block, not soft.** No partial read-only mode — half-joining contradicts the platform pitch. Search-engine crawlers (Googlebot, Twitterbot, etc.) bypass the splash so SEO/OG previews work. **Surface differentiation, not platform-branded copy:** Android shows a real `[ Open in browser ]` button that fires an intent link; iOS shows NO button (Apple won't allow programmatic redirect to Safari) — instead, a static inline tip *"Tap Share, then 'Open in Browser'"* with share icon. Same headline + body across both platforms. **Why it matters:** in-app browsers have isolated localStorage, sometimes wipe between sessions, and never support push notifications. Letting users in creates phantom identities they will lose. **Anti-patterns:** allow-listing only known-good browsers (false-positives on niche browsers), soft warning that lets through (creates phantom identities), JS redirects to Safari (Apple killed every reliable trick years ago). See LAUNCH_PLAN.md D2.

- **Mobile modals adopt the AgentChat bottom-sheet pattern (no library) (settled 2026-05-10).** Six modals (`SignInModal`, `IdentityBar` You modal, `MoveAddressModal`, `RestoreModal`, `ChangePassphraseModal`, `FundAddress`) restructure to bottom-sheet on mobile / centered on desktop using the **same Tailwind pattern already proven in `src/app/AgentChat.tsx:207`**: container `fixed inset-0 z-[…] flex items-end sm:items-center justify-center sm:p-4 pointer-events-none`, panel `w-full sm:max-w-{sm|md|lg} rounded-t-2xl sm:rounded-2xl pointer-events-auto animate-[slideUp_0.3s_ease-out]`. **Wizards** (`MoveAddressModal`, `RestoreModal`, `ChangePassphraseModal`) get full-height sheets via `min-h-[80vh] sm:min-h-0 overflow-y-auto`. **Single-step modals** (`SignInModal`, `FundAddress`) get the half-height treatment. **`IdentityBar` (You modal) needs `max-h-[92vh] overflow-y-auto`** — it's the tallest modal (earnings + chart + activity + balance + rows) and overflows on small phones without the height cap. All preserve their existing internal logic, including the locked-state You modal cross-fade (DECISIONS.md 2026-05-01). **Anti-patterns:** don't introduce vaul / react-modal-sheet / any external library — the in-house pattern works; don't break the desktop centered-popover pattern; don't break the locked-state cross-fade. See LAUNCH_PLAN.md D3 + per-modal class table.

- **QR device sync is a post-launch layer, not launch (settled 2026-05-10).** Recovery file remains the structural primitive for cross-device identity transfer. QR sync is a UX accelerator that sits on top, planned as Bucket 6 after launch. Cryptographic model: source device wraps its already-encrypted `bfn_keypair_enc` blob in a one-time AES-GCM envelope (random transfer key), uploads envelope to server with short ID, ID + transfer key go in the QR (key in URL fragment so server never sees it), envelope auto-deletes after 5 min, destination scans QR, decrypts envelope locally, prompts user for passphrase. Plaintext WIF NEVER leaves source device — two layers of encryption, three independent factors required for compromise (physical access to QR, network access ≤5 min, passphrase). **Plaintext-mode users blocked from sync** until they set a passphrase — forcing function for the right behavior. **Five required cryptographic deltas before code** (per code-auditor 2026-05-09): use raw-key AES-GCM helper not PBKDF2 wrapper; one-shot atomic delete-on-read on GET endpoint; 5-attempt cap on typed-code fallback (single most important security control); bind envelope to id + origin via AES-GCM AAD; destination validates inner blob shape + ships full JSON wrapper + prompts-with-auto-backup before overwrite. See LAUNCH_PLAN.md D7 + Bucket 6 for full spec.

- **Single-broadcaster proxy at launch with miner-agnostic result type (settled 2026-05-10).** The `/api/broadcast` proxy ships configured to use ONLY GorillaPool at launch — no TAAL fallback yet. **Architecturally critical guardrail:** the broadcast helper exposes a single `broadcast(rawTx)` function with miner selection encapsulated inside; the result type is `{ status, txid, code }` — miner-agnostic. **Don't leak `gorillapool_*` strings into call sites or status enums.** With this guardrail, the double-broadcast hazard (an ARC success-then-CDN-failure mistakenly triggering TAAL retry) never materializes because only one broadcaster exists, and adding TAAL later is a localized internal change. Without the guardrail, adding fallback later becomes a cross-cutting refactor. Trade-off accepted at launch: an ARC outage freezes posts/boots until recovery (small launch-window blast radius given GorillaPool's track record). **When TAAL is added later:** classification rules required — `200/201` → success; `5xx` with no body → fallback to TAAL; `5xx` WITH ARC error body containing `txid` → return success-shaped response (tx accepted before HTTP layer failed); on timeout, query primary's `/v1/tx/{txid}/status` for 2s grace before falling back. Proxy is byte-transparent on request body — does not parse or transform (BEEF ancestry passes through untouched). See LAUNCH_PLAN.md C1 for full spec.

- **Two protection helpers, not one (settled 2026-05-11).** Identity service exports two helpers with deliberately different semantics that look almost identical and will be tempting to collapse: `isIdentityEncrypted()` checks ONLY whether an encrypted store (`bfn_keypair_enc`) exists; `isEffectivelyProtected()` returns true ONLY if encrypted exists AND no plaintext key exists. The distinction matters because of the **interrupted-upgrade case** — if a passphrase upgrade crashed mid-flow before `commitUpgrade()` could remove the plaintext, both keys remain in localStorage. `getIdentity()` already correctly prefers the plaintext one in that case, so the user is *effectively unprotected* (plaintext is what's actually being used). **UI gating must use `isEffectivelyProtected()`**, NOT `isIdentityEncrypted()` — otherwise unprotected users get routed to a passphrase prompt for a passphrase they never successfully set up. `isIdentityEncrypted()` is reserved for the internal race-handling inside `getIdentity()` where the narrower presence-check semantics are required. **Do not collapse these into one function.** See `src/services/bsv/identity.ts` doc comment.

- **iCloud Keychain requires a username anchor on rotation forms (settled 2026-05-18).** iOS Safari's "Save Password?" / "Update Password?" sheet only fires when the form has a stable `autocomplete="username"` value that iCloud can use to key the credential lookup. Without it, iOS skips the save prompt entirely on subsequent rotations — the first save works (no prior entry to match), but every rotation after that is silently ignored because iOS can't tell whether this is a new credential or an update to an existing one. Verified on iPhone iOS 18 (user saw exactly one saved entry in Settings → Passwords for bsvibes.com, regardless of how many rotations were performed). **Fix:** wrap the password inputs in a `<form>` element with a hidden `<input type="text" autoComplete="username" value={identity.name} readOnly hidden />` as the first child. `identity.name` is stable across rotations (the on-chain migration protocol preserves the name), so iOS sees subsequent rotations as updates to the same credential and fires "Update Password?" with the new value. Source: WWDC 2018 session 204 + Apple Password AutoFill docs + WHATWG autocomplete spec. **Do not use a value that changes across rotations (e.g. the hint).** That would defeat the matching logic and create duplicate keychain entries on every rotation. **Do not omit the username field.** That returns iCloud to silent-fall-through behavior.

- **Block scope covers both pagehide AND visibilitychange (settled 2026-05-18).** E24 introduced `blockSessionClear()` to suppress `pagehide → clearSessionCaches()` during iOS system sheets on PWA. **It was incomplete.** The iOS Save Password sheet (and Share, Files picker, etc.) fires BOTH `pagehide` AND `visibilitychange→hidden` on a standalone PWA. The `visibilitychange` handler in `IdentityBar.tsx` (which sets `manageAuthed=false` to destroy the manage gate on tab blur) was unprotected by E24 and fired during the iOS sheet, cascading through React re-renders and effectively closing the rotation modal mid-flow. **Fix (E26):** IdentityContext exposes `isSessionClearBlocked(): boolean` reading the same ref-counted block. IdentityBar's visibilitychange handler short-circuits when the block is active. One ref, two consumers — keeps the suppression surface coherent. **Any future visibility-driven teardown anywhere in the app must read `isSessionClearBlocked()` first.** Otherwise we'll re-introduce this class of bug.

- **Local state commits BEFORE parent notification in rotation/restore flows (settled 2026-05-18).** In React 19.2 + React Compiler, when a child component (like MoveAddressModal or RestoreModal) needs to render a "done" state AND notify the parent that identity has changed, the call order matters. Calling `onComplete(newIdentity)` first triggers `updateIdentity()` in the parent, which causes a parent re-render BEFORE the child's local `setStage("done")` / `setImportSuccess(true)` commits. The reconciliation can unmount the child before the done state renders. **Fix:** always commit local done-state FIRST, then notify parent. In MoveAddressModal `runRecording()`: `setStage("done")` → `onComplete(result.identity)`. In RestoreModal `performImport()`: `setImportSuccess(true)` → `onSuccess(imported)`. The bug-class generalizes — any modal that has its own dismissal/done state AND fires a state-mutating callback to the parent must commit its own state first. **Do not reverse this order.**

- **No auto-close timers on success states in restore / rotation modals (settled 2026-05-18).** RestoreModal previously had `setTimeout(() => { onSuccess(); handleClose(); }, 1200)` — fired 1.2s after success regardless of any iOS sheet timing or user attention. This caused the modal to close before users could see the confirmation that their restore had succeeded, especially on PWA where the iOS Save Password sheet sat on top. **Fix:** all rotation/restore success states transition to an explicit "done" step with a user-dismissable button (Got it / Continue). Parity with ChangePassphraseModal / MoveAddressModal done-states. **Do not re-introduce auto-close timers** — they always race with iOS system sheets and either confuse the user or hide critical confirmation UI.

- **Restore-from-encrypted-file adopts the file's passphrase as the new identity's passphrase (settled 2026-05-22).** When a user restores from a passphrase-protected recovery file, the passphrase they typed to decrypt the file becomes the passphrase guarding their new identity going forward. The file's hint is preserved too. **Fix:** `importEncryptedIdentity(wif, passphrase, name?, hint?)` in `identity.ts` re-encrypts the restored WIF and writes it to `bfn_keypair_enc`. Mirrors `upgradeIdentity`'s store shape so `getStoredAnonName` / `getStoredHint` continue to work. Session caches (`_sessionIdentity` etc.) are primed inside the function so `signPost` (used immediately by `cleanupMigrations`) succeeds without forcing the user to re-unlock — the passphrase was JUST provided. **Behavior before this:** restore wrote plaintext via `importIdentity`, discarding the typed passphrase. User had to rotate again to be protected. The "passphrase adopted on restore" pattern was NEVER in git history before this — confirmed via `git log --all -S "encryptWif"`. Do not assume it's a regression restoration; it's a new feature. **Do not collapse `importIdentity` and `importEncryptedIdentity` into one function** — they have distinct semantics (plaintext write vs encrypted write).

- **No auto-download on rotation; explicit Save with stakes context (settled 2026-05-22).** Pre-E27 rotation auto-downloaded a recovery file the moment `runRecording` completed. This produced UX friction (iOS PWA full-page download sheet on every rotation, multiple files piling up in Downloads with similar names) and an authenticity problem (`markBackedUp()` fired automatically, falsely claiming the user had saved when iOS may have actually dropped the download). **Fix:** `MoveAddressModal.runRecording` no longer calls `downloadBackup`. The combined backup payload is held in `combinedBackupRef`. The done-state UI is now a context card showing the user's earnings (sats + USD via `useBsvPrice`) with an explicit "Save recovery file" primary button and "I'll do it later" secondary link. Tapping Save uses the Web Share API; tapping Skip leaves the new key without an external file but the amber "Unsaved key" warning dot persists in IdentityBar via the per-address saved flag. **Disaster-recovery preserved:** pre-rotation file STILL auto-emits on failure mid-flight (the safety net is in the failure path, not the happy path). **Do not re-add the auto-download in `runRecording`.** **Do not call `markBackedUp()` from `onComplete` (rotation reached done stage)** — it must only fire after the user actually saves. Use the new `onSaved` callback prop on MoveAddressModal for that signal.

- **Web Share API for recovery files; AbortError is a deliberate cancel (settled 2026-05-22).** `shareOrDownloadBackup(data)` in `backup-template.ts` is the new save path. It uses `navigator.share({ files: [file] })` when available, giving iOS users the familiar rounded share drawer with "Save to Files" as a top-level option, instead of the intrusive full-screen download sheet. Implementation constraints (per Web Share API spec + community reports):
  - **MIME type is `text/html`** (E28a, 2026-05-25 — superseded the earlier E27 decision to use `application/octet-stream`). E27 chose octet-stream based on a researcher's "iOS treats text/html as hostile" guidance, but iPhone PWA testing showed the share drawer NEVER appeared at all — silent fallback to `<a download>` every time. Current research + community consensus: WebKit's PWA process uses a stricter file-MIME allow-list than the Safari tab process, and `application/octet-stream` is OFF the PWA list while `text/html` is ON it. Diagnostic logs added in E28a will confirm; will revert if disproven. **Do not flip back to octet-stream without first proving via diagnostic logs that text/html is the cause of share failure**, since the share drawer working on PWA is one of the central UX wins of E27.
  - **Build the `File` object SYNCHRONOUSLY inside the click handler before any `await`** — iOS Safari's transient activation token expires across async boundaries. The function does the entire HTML generation + File construction sync before calling `navigator.share`. Callers must NOT `await encryptWif` or similar between the click and invoking the helper; pre-build any encrypted payload in a `useEffect` and stash it in state.
  - **`AbortError` is the user dismissing the share drawer.** Return `{shared: false, cancelled: true}` and DO NOT fall back to `<a download>` — that would re-trigger the intrusive download sheet the user just deliberately closed. UX regression.
  - **Other share errors fall back to `<a download>` via the legacy `downloadBackup`.** `canShare` can return true and `share` still throw — always wrap in try/catch.
  - **Caller is responsible for `blockSessionClear`** — the iOS share sheet fires `pagehide` and `visibilitychange→hidden` like any other system sheet, so callers should hold the session-clear block across the share. See `MoveAddressModal.handleSaveBackup` and `RestoreModal.handleSaveOldKey` for the pattern.

- **Per-address saved flag — global backedUp + per-address bsvibes_saved:&lt;addr6&gt; (settled 2026-05-22).** Pre-E27 a single global `bsvibes_identity_backed_up` boolean tracked "has the user ever saved a recovery file." Post-E27 we layer a per-address flag (`bsvibes_saved:<addr6>`, ISO date) on top. Each rotation creates a new address; the global flag remains true (the user has SOME saved file in their history) but the new address is freshly unsaved. The amber "Unsaved key" warning dot in `IdentityBar` reads `backedUp === false || !isAddressSaved(identity.address)` — either condition surfaces the dot. **The global flag MUST ONLY flip on explicit user save** — moved out of `MoveAddressModal.onComplete` and into the new `onSaved` callback that fires from `handleSaveBackup` after successful share. Per-address keys also written by RestoreModal `performImport` (because the file the user just restored IS the backup) and by all existing `markBackedUp()` call sites (Save/Copy/Show key paths in IdentityBar). **Do not collapse the two flags** — they have distinct semantics.

- **RestoreModal restore-pre is Skip-with-confirmation-toggle (settled 2026-05-22).** Pre-E27, when a user restored to a new identity, the modal auto-downloaded their OUTGOING identity's recovery file before swapping (defense-in-depth). Post-E27, this is a user choice: a context card shows the outgoing identity's stats + Save (primary) and Skip (secondary). Tapping Skip transitions to a red warning state requiring a second tap on "Skip & restore anyway." Force-save was rejected — users with their backup already saved elsewhere (in a password manager, another file, etc.) shouldn't be forced to re-download. The two-step skip provides informed-consent without total prevention. **Security note:** unprotected identities with no externalised key WILL be permanently lost if the user Skips. Acceptable because the user explicitly confirmed the warning. **Do not collapse the two-step Skip into a single button** — the friction is the consent gate.

- **Restore of rotated keys is blocked outright (settled 2026-05-26, E29 — "Design C-strict").** Any recovery file whose key has a forward migration record on-chain CANNOT be restored as an active identity. The on-chain migration is treated as a permanent revocation event — like Google / Apple invalidating device tokens on a password change. This closes a critical security vector: every OpenCook user's FIRST identity is plaintext by default (no passphrase until they upgrade), so the plaintext recovery file is a non-revocable leak vector. If restore-then-reclaim were allowed (the previous behavior via `cleanupMigrations` auto-rewrite), anyone who ever obtained the plaintext file (laptop theft, email leak, cloud trash, screenshot) could later take over the user's future earnings — even years after they upgraded to a strong passphrase. The block uses the migration record as the only source of truth: once a user rotates, the old key is permanently retired from OpenCook's UI perspective.

  **Implementation:** new read-only endpoint `GET /api/restore-eligibility?pubkey=<hex>` returns `{ allowed: true }` if no forward migration exists, else `{ allowed: false, rotatedAt, newAddrPrefix }`. Rate-limited 30/min/IP. The check fires BEFORE any identity-state write in both `RestoreModal.doImport` and `HomeScreenWelcomeGate` (both `handleFile` plaintext branch + `handlePassphrase` encrypted branch). Fail-safe: any network/parse failure during the check ALSO blocks — without verification we can't safely allow the restore.

  **The error UX is non-blocking** — the user is told their key was rotated, given the date + new address prefix, and pointed at finding their most-recent recovery file. They can also use the WIF in an external BSV wallet (Yours, Centi, Sparrow) to spend any UTXOs at the old address — OpenCook is not a custodian of those funds.

  **Trade-off accepted:** users who lost their newer key and only have a pre-rotation file cannot recover via OpenCook UI. Mitigated by the existing combined-recovery-file pattern: every rotation file contains BOTH keys under one passphrase, so only the FIRST plaintext save (before any rotation) is lost-recovery-blocked.

  **`cleanupMigrations` server action REMOVED in E31 (settled 2026-06-01).** Originally added 2026-03-28 (commit `31a9d92`) to fix a specific bug: a user could rotate A→B, lose B, re-import A, and then have every future payout silently redirect to the (now lost) B address. Re-importing A used to be permitted; `cleanupMigrations` deleted the stale A→B migration row so payouts flowed to A again. **E29 (2026-05-26) made that scenario structurally impossible** by blocking re-import of any rotated key — so the function's only valid call site (RestoreModal's `doImport`) was already removed, and the underlying bug it fixed can no longer occur. The function then sat orphaned in `actions.ts` with a documented retention rationale ("future signature-gated admin reclaim design") that didn't survive scrutiny: that future design would need different auth (admin pubkey, not the from-key holder's WIF), different message format, and an audit log — making the existing 75 LOC reference material at best, not literal reuse. With E31 closing the rotate-from-stale takeover, keeping `cleanupMigrations` as a guarded-but-still-callable server action was the worst of both worlds (dead code that's still importable). **Recoverable from commit `31a9d92` (`git show 31a9d92:src/app/actions.ts`) if a future admin-reclaim feature ever materialises.**

  **DO NOT add a UI path that allows restoring rotated keys**, even with a warning. The previous architect-recommended "warn + opt-in reclaim" hybrid was rejected — it left the same plaintext-key attack vector open, just behind a click the attacker happily makes. **DO NOT re-introduce `cleanupMigrations` without an admin-signature auth model.** **DO NOT remove the migration-record check** — the security model depends on it.

  Edge case: a key previously reclaimed via the legacy `cleanupMigrations` auto-bridge AND THEN re-rotated will be permanently blocked from restore. Acceptable — user can recover their UTXOs via external wallet; rotation forward is irreversible by design.

- **E30 stale-key session-lockout (settled 2026-05-28, "UI-layer only").** When a device holds a key that has been rotated forward on-chain (typically: user rotated on another device and then opens this one), the app surfaces a `<StaleKeyModal>` on first detection and replaces the textarea in `PostForm` with an amber banner blocking the compose flow. This is a UI-layer lock — `createPost` / `bootPost` server actions are NOT modified and still accept signatures from the old key (which is the deliberate trade-off; see SECURITY_AUDIT.md L7).

  **Chosen approach:** detection via polling. `useFeedPolling` sends the local pubkey in an `x-bsvibes-pubkey` request header on every `/api/posts` poll (header, not query string — keeps pubkey↔IP correlation out of CDN/proxy URL access logs). The server endpoint returns `key_status: { stale: true }` only when ALL of (a) `process.env.E30_STALE_KEY_ENABLED === "true"`, (b) the header passes shape validation, (c) a forward-migration row exists. Any other condition omits the field. Client treats anything other than strict-equal `{ stale: true }` as not-stale (F1+F2 fail-open — a server bug must never mass-lock users).

  **State machine:** new `kind: "staleKey"` variant on `IdentityState` (`useIdentity.ts`). The underlying identity is retained in `staleKey` state so the chip can show the cached anon name and non-signing reads (address, name, pubkey) keep working; mutation handlers gate via `requireIdentity()`, which now branches: stale → `openStaleKeyModal()` instead of `openSignIn()`. The context wrapper `markIdentityStale` is suppressed when `isSessionClearBlocked()` is true (F3 mitigation — prevents self-stale during own-device rotation when MoveAddressModal/ChangePassphraseModal hold the block).

  **Reasoning for UI-only vs per-mutation server gating:** open/closed principle. E29's `/api/restore-eligibility` is already proven; E30 reuses the same `getForwardMigration` helper for the polling check and the existing `requireIdentity()` Hard Rule #7 universal pattern for the gate. Any future mutation feature (boost, tip, etc.) inherits the lock automatically — no per-feature server changes needed. Lower LOC, no new server attack surface. A malicious WIF holder bypassing the UI entirely (direct BSV broadcast) is documented as L7 residual risk with a clear retreat path (add `getForwardMigration` lookup to `createPost`).

  **U1 escape hatch** ("I don't have the newer file"): inline expand-below within `<StaleKeyModal>`. Tells the user honestly that earnings + posting follow the newer key, on-chain history is intact, the older key on this device can no longer post or earn. No "support" hook, no recovery promise, no "start fresh" path. Acceptable per the threat model — frequency is low (rotated + lost newer file + still has older device).

  **R1 fix:** IdentityBar Show Recovery Key panel subscribes to `staleKey` and force-closes the dropdown on transition — otherwise a user mid-reveal could photograph/copy a now-dead WIF.

  **Auto-open behavior:** the StaleKeyModal auto-opens on the FIRST `ready → staleKey` transition (via a ref-guarded effect in IdentityContext). Subsequent polls that re-fire `markIdentityStale()` are no-ops at the hook level (already-stale state), so a user-dismissed modal does NOT spontaneously reopen. The amber banner is the persistent re-entry point.

  **Z-index stacking:** SignInModal z-[80], StaleKeyModal z-[90], RestoreModal z-[100]. RestoreModal was bumped from z-[70] in E30a so the restore flow can stack cleanly on top when chained from the stale modal.

  **Pre-rotation pubkey backfill:** `Identity` type gained a required `pubkey: string` field in E30a; `materializeFromStored` derives it on load for pre-E30a `StoredIdentity` payloads and rewrites the localStorage entry so subsequent loads skip the SDK round-trip. Backfill is best-effort (write failures don't block return).

  **Retreat path:** if attribution griefing is observed (L7 escalates), add a `getForwardMigration` check inside `createPost` server action and reject — one-session change. UI gating stays as defense-in-depth, server gating adds the second layer.

  **DO NOT remove the F3 guard** in `IdentityContext.markIdentityStale` — it prevents self-stale during own-device rotation. **DO NOT remove the fail-open posture** in `shouldCheckStaleness` / `useFeedPolling` — strict-equal `{ stale: true }` is the only signal that triggers the lock. **DO NOT switch pubkey from header to query string** — the privacy concern is real even though pubkeys are public on-chain (header keeps the correlation out of default proxy logs).

- **E31 block rotate-from-stale (settled 2026-06-01, "symmetric to E29").** Discovered during E30 manual testing: a stale-key holder could still call `migrateIdentity` to rotate their key. Old WIF signs a valid migration → server accepts (signature is cryptographically correct) → `INSERT OR REPLACE` on `migrations` table silently overwrites the legitimate rotation → chain head moves to the attacker's new key → legitimate current key holder gets locked out. Same attack class as E29 (which closed restore-from-stale), just at a different endpoint.

  **Server-side fix:** `migrateIdentity` now calls `getForwardMigration(oldPubkey)` after signature verification and rejects with `reason: "stale_key"` if a forward migration row exists. Same pattern as E29's `/api/restore-eligibility` check. Fails CLOSED on DB lookup errors (unlike E30's poll-time fail-open posture — re-opening the rotate-from-stale bug to a partial DB outage would defeat the purpose).

  **Client-side preflight:** both `MoveAddressModal.runCreating()` and `ChangePassphraseModal.handleChange()` now call `/api/restore-eligibility` BEFORE invoking `upgradeIdentity` (which runs the sweep). Without the preflight, the sweep would move funds to a new address that the server-side `migrateIdentity` then rejects — leaving funds at an address the server can't recognise (user still has the new WIF so funds aren't lost, but the OpenCook UI state is broken). Reuses E29's endpoint — same semantic question ("is this pubkey eligible to be the active key going forward?").

  **MoveAddressModal return-value check:** previously `await migrateIdentity(...)` was fire-and-forget — same class regression as historical BUG-10 (fixed in ChangePassphraseModal but never patched in MoveAddressModal when the latter was introduced). E31 fixes this: result is checked, stale-key reason surfaces specific copy, other failures throw generically.

  **UI trigger guard:** `IdentityBar.openMoveModal` now intercepts when `staleKey === true` and calls `openStaleKeyModal()` instead of mounting the rotation wizard. Three call sites feed through `openMoveModal` (Passphrase row, Not Protected red banner, manage-gate fallback) — guarding once covers all of them. Defense-in-depth on top of the server gate + client preflight.

  **Decisions made during design:**
  - Hard-lockout gate (replace feed with full-screen restore prompt) was considered and rejected — both UX and architecture agents independently recommended against. Reasons: punishes the legitimate multi-device case, removes earnings/feed visibility while stale, removes the persistent amber banner as a re-entry signal, would require relocating `useFeedPolling` outside FeedContent to avoid polling-stops-while-stale. The "stale-aware surfaces" count is 4-5 (well under the inflection point of ~6 where hard-gate becomes the right call). E30's modal+banner pattern stays.
  - `cleanupMigrations` deleted entirely rather than guarded (see updated entry above).

  **DO NOT remove the server-side `getForwardMigration` guard** in `migrateIdentity` — the entire E31 fix depends on it. **DO NOT switch the server-side check to fail-OPEN on DB errors** — that re-opens the takeover. **DO NOT remove the client-side preflight** — without it the sweep can run before the server rejects, stranding funds at an address the server doesn't recognise.

- **No `title` field in `navigator.share` when sharing files (settled 2026-05-25).** Earlier E27 implementation passed `navigator.share({ files: [file], title: "OpenCook recovery key" })`. iPhone Safari testing showed this saves the HTML recovery file AND a sidecar `.txt` file containing just the title string "OpenCook recovery key". Known iOS Web Share API quirk: when both `files` and `title` are present, iOS treats them as separate share items. **Fix:** share files only — drop the `title` field. **Do not re-add `title` to file-sharing calls.** If a label is needed for the user, communicate it in the UI before invoking share, not in the share payload itself.

- **iOS Quick Look noscript / input-readonly pattern (settled 2026-05-18).** Two iOS-Quick-Look-specific quirks force the recovery file HTML to be written in a non-obvious way that contributors will be tempted to "fix" back: (1) `<noscript>` content does NOT render in iOS Quick Look even though no JavaScript executes. The HTML spec says `<noscript>` renders when the *browser engine* reports scripting as "disabled," but Quick Look's sandboxed WebKit still reports scripting "enabled" at engine level — it just never runs scripts. So `<noscript>` content stays hidden. **Solution:** use a normal `<div id="quicklook-notice">` visible by default, hidden by a tiny JS IIFE on load. The "JS-required" warning shows in Quick Look (no JS) and disappears in real browsers (JS runs and hides it). (2) `user-select: all` is unreliable in iOS Quick Look — the preview UI intercepts touches at a layer above CSS, so long-press-to-copy on a `<div>` or `<span>` doesn't fire. **Solution:** use `<input type="text" readonly>` for the address row and `<textarea readonly>` for the WIF — native form controls retain OS-level tap-to-select / long-press handles in Quick Look. Pattern used by 1Password Emergency Kit, Bitwarden emergency sheet. **Do not revert to `<noscript>`. Do not switch addresses or WIFs back to spans/divs.** CSS strips native form-control defaults so inputs look identical to today's spans. `copyText()` reads `el.value` for form controls and `el.textContent` for spans (the "Saved" date row is still a span). Verified by the researcher agent against MDN spec + community reports + Apple's Quick Look docs (which don't explicitly document this behavior, but the spec mechanism + industry consensus is consistent).

- **iOS Permissions API for "microphone" is unreliable; call recognition.start() directly (settled 2026-05-16).** Tested on iPhone Safari: `navigator.permissions.query({ name: "microphone" })` returns a cached `denied` state long after the user re-enables mic access in Settings → Safari → Microphone. The cache only refreshes on hard refresh / app reinstall. **Don't gate `recognition.start()` behind a permissions pre-check** — it surfaces the native iOS prompt on first use and the existing `onerror` handler catches genuine `not-allowed` errors. A previous version of `PostForm.tsx` had this pre-check and it caused our "Enable in Settings → Safari → Microphone" toast to fire forever even with permission granted. The pre-check is both redundant and broken — recognition.start() is the single source of truth.

- **autoComplete attributes are required on all password inputs (settled 2026-05-16).** iOS 17+ Safari and iCloud Keychain only trigger the "Save Password?" prompt when `<input type="password">` carries the correct `autoComplete` value. **Use `autoComplete="current-password"`** on unlock paths (SignInModal, IdentityBar manage gate, ChangePassphraseModal verify step) and **`autoComplete="new-password"`** on rotation paths (ChangePassphraseModal new + confirm, MoveAddressModal new + confirm). Without these, Safari treats the field as a generic password input and skips the save prompt — iCloud Keychain never stores the passphrase. Verified: a Safari user reported the save prompt had silently stopped working; root cause was that none of our 7 password inputs had `autoComplete`. **Any new password input must carry an autoComplete attribute.**

- **Pagehide-driven session clear must be blockable during rotation flows (settled 2026-05-16).** `IdentityProvider` listens for `pagehide` in standalone PWA mode and calls `clearSessionCaches()` — the intentional password-manager-style cleanup that fires when the user really backgrounds the app. iOS also fires `pagehide` on its own *system sheets* — Save Password, Share, Files picker — which is NOT a real backgrounding event but our handler can't tell the difference. Without protection, the iOS Save-Password sheet shown after a passphrase rotation would torch the in-memory session and silently unmount the modal before the user saw "Download again / Got it". **Solution:** ref-counted `blockSessionClear()` / `unblockSessionClear()` on IdentityContext. The pagehide handler checks the ref before clearing. ChangePassphraseModal blocks at `handleChange` entry and unblocks in `handleClose`; MoveAddressModal blocks at `runCreating` entry and unblocks via a wrapped `onClose` funnelling every dismissal path. Both have a useEffect cleanup as a mid-flow unmount safety net. **Any future flow that triggers an iOS system sheet during sensitive identity state must wrap itself in block/unblock.**

- **Welcome gate fires on standalone-mode + no identity, four real causes (settled 2026-05-10).** When the app opens in PWA standalone mode AND no identity exists in localStorage, the welcome gate fires asking the user to restore from a recovery file or start fresh. The four real causes that produce this state, all of which need honest copy: (a) **iOS 7-day ITP eviction** — Apple's automatic privacy feature wipes site data after ~7 days of no visits, even from "properly installed" Safari PWAs; (b) **iOS Private Browsing install** — storage is ephemeral, gate fires every cold open (rare); (c) **Non-Safari iOS install** (Brave/Chrome iOS) — different storage sandbox from Safari; (d) **Fresh install / manual storage wipe**. **Copy must be honest in all four cases:** *"We couldn't find your identity on this device"* — reads as a normal restoration prompt, not a system error. **Don't promise** *"you only see this if you installed from a non-Safari browser"* — would read as a bug when ITP eviction kicks in. **Implementation guardrail:** synchronous pre-hydration check in `IdentityProvider`'s lazy initializer must defer auto-key-generation until welcome-gate decision; race condition otherwise where IdentityProvider auto-generates a fresh key before WelcomeGate mounts. Drop the "first-launch flag" entirely (would persist across storage wipes and cause gate-fires-only-once-ever). See LAUNCH_PLAN.md C3, #3, #5.

- **Save → Install → Welcome-gate → Notifications is one ordered flow (settled 2026-05-10).** The trigger sequence is: (1) silent identity creation on first visit — no nags; (2) after meaningful action (first earning, first boot, or explicit "Save" tap), the save flow produces a recovery file; (3) ONLY THEN does the install pitch fire; (4) user installs; (5) on the first home-screen open the welcome gate resolves identity; (6) notification permission is requested after the gate completes. **Ordering rule:** the install pitch MUST NOT fire until all four conditions hold: `recoveryFileExists() && !isStandaloneMode() && isSupportedInstallPlatform() && !installPitchDismissedRecently()`. This logic lives in `src/lib/install-pitch.ts`. Implemented in `src/components/InstallPitch.tsx`, `src/contexts/InstallContext.tsx`. **Why the order matters:** on iOS, installing creates a new storage sandbox that does not share localStorage with the browser tab that existed before install. If the install pitch fires before a recovery file exists, the user installs, lands in a fresh sandbox with no identity, and has nothing to restore from. The ordering guarantee means any user who reaches the install pitch has a recovery file and therefore a working restoration path. **Do not fire the install pitch on first visit**, on any visit before a file exists, or after "more secure" framing — install does not improve security, only convenience and privacy. **Do not remove or relax the `recoveryFileExists()` gate** without a replacement that provides equivalent restoration confidence. See LAUNCH_PLAN.md D4, Status #10.

- **Welcome-gate copy uses intent, not technical state (settled 2026-05-10).** The home-screen first-launch screen is: header *"Welcome back or starting fresh?"*, body sentence *"We couldn't find your identity on this device."*, primary button *"Restore from your saved file"* (sub-text: *"Use your most recent recovery file. Your posts and earnings come back."*), secondary button *"Start with a new identity"* (sub-text: *"Begin fresh on this device. You can save and restore later."*). This copy was arrived at after deliberate review — **do not replace it with technical-state descriptions.** Shipped in `src/components/HomeScreenWelcomeGate.tsx`. **Why intent-framing:** users have no reason to understand iOS storage sandbox mechanics, ITP eviction, or PWA install isolation. "Returning or starting fresh?" maps cleanly to every real cause that lands a user on this screen (non-Safari install, ITP eviction, Private Browsing install, manual wipe) without needing per-cause branching. Auto-detecting the "right" choice creates false confidence — the user knows their intent better than a localStorage heuristic. "Start fresh" must not feel penalised; it's a valid choice for new-device-first-run users. The "Scan from another device" tertiary option is appended here once Bucket 6 QR sync ships. **Do not** auto-select a path, add technical-state language like "if your address has changed…", or make "Start fresh" feel scary. See LAUNCH_PLAN.md D5, Status #3, #5, #9.

- **Service worker scope discipline: push-only, no fetch handler (settled 2026-05-10).** *(Forward-looking guardrail — the SW does not exist yet. `public/sw.js` is NOT in the repo as of 2026-06-04. This entry locks the design for when Bucket 3b lands.)* The OpenCook service worker (Bucket 3b) is **push-only**. It registers a push subscription, handles `push` events, and nothing else. **There is no `fetch` event listener. There are no caching strategies. There is no offline support.** The app requires a live network connection to read the chain and SQLite — offline mode is architecturally incoherent for this product. Caching strategies in a Turbopack dev environment produce non-deterministic SW behavior that is hard to debug. **Do not add a fetch handler to the service worker**, even for static assets. Do not add `workbox`, `sw-precache`, or any library that implies caching. Do not treat the SW as a background-sync surface for posting; that belongs in a dedicated Phase 7 architecture decision. The SW file is `public/sw.js` and must be registered from a CLIENT component (not from a server layout boundary — see C5 entry). CSP requires `worker-src 'self'`. **The one exception:** a minimal `install`/`activate` lifecycle handler to claim clients is acceptable and expected. If a PR adds ANY `fetch` or `cache` operation to the SW without an explicit supersession of this decision, treat it as an accidental regression. See LAUNCH_PLAN.md D6, C5.

- **Notification copy discipline: single-line present-value, no roadmap hedging (settled 2026-05-10).** *(Forward-looking — push notifications and the `NotificationPrompt` component are Bucket 3b and not yet built. - **Install pitch surfaces — no timer-based dismissal (revised 2026-06-03, supersedes 2026-05-10).** The install pitch has three surfaces, each appearing only when the 5-condition `shouldShowInstallPitch` gate passes (`backedUp && protected && !standalone && installType supported && !engaged`):
  - **Slide-up sheet** (`InstallPitch variant="banner"`) — auto-fires once per tab session ~800ms after the gate opens. Full-screen modal with backdrop, 64px app icon, "Get the APP experience" headline, platform-appropriate CTA. **No X button** — only a chevron-down at the top center that minimises to the bookmark. Backdrop click also minimises.
  - **Bookmark** (`InstallBookmark`) — 20px OpenCook app icon button in PostForm's footer row (centered between helper text and Ask AI pill). Tap re-opens the sheet. Fires Ask-AI-style highlight flash on sheet→bookmark transition (2s).

Save-file → upgrade-to-passphrase → install. Plaintext-key users see the "Not protected" red banner instead, pushing them to upgrade first. Do NOT add an X to any surface — minimise to bookmark is the dismiss model. Do NOT replace the inline row's whole-row-click-opens-sheet with an inline install button (creates nested-interactive HTML and inconsistent platform behavior). Do NOT remove the modal-block ref-counter — without it, the sheet ambushes the user mid-rotation. - **Standalone-mode detection uses three signals plus a reactive listener (settled 2026-05-10, architect's C4).** `useStandaloneMode()` (`src/hooks/useStandaloneMode.ts`) detects installed PWA mode via all three of: (1) `window.matchMedia("(display-mode: standalone)").matches` — the W3C standard signal, also fires for `minimal-ui` and `fullscreen` manifest display modes; (2) `navigator.standalone === true` — iOS Safari legacy property, required because iOS 16.3 and earlier do not fire the `display-mode` media query reliably in the initial render; (3) a 30-day `bsvibes_installed` localStorage flag set on `appinstalled` event capture (Android Chrome fires this event on successful install). Additionally, the hook attaches a `matchMedia` change listener on the `display-mode` media query to reactively update when the user exits/enters Stage Manager or Split View on iPad. **Do not collapse to a single `navigator.standalone` check** — that is iOS-only and has the reliability caveat above. **Do not collapse to a single media query** — pre-iOS-16.4 installations are missed. All three signals must remain. The `appinstalled` flag is an additive belt-and-suspenders that prevents the install pitch from re-firing in the session immediately after install while the other signals catch up to DOM state. See LAUNCH_PLAN.md C4, D4.

- **Service worker integration with Next.js 16: four required guardrails (settled 2026-05-10, architect's C5).** When Bucket 3b lands, the service worker must be integrated following these constraints — all four are required; omitting any will produce either a broken Turbopack dev build, a broken iOS install flow, a CSP violation, or an insecure push subscription: (1) **No fetch listener.** See the "service worker scope discipline" decision above. In Turbopack dev mode, a fetch handler in the SW creates caching conflicts with Turbopack's own asset-serving pipeline and produces unreliable HMR. (2) **Register from a CLIENT component, not from `src/app/layout.tsx`.** `layout.tsx` is a Server Component. Service worker registration via `navigator.serviceWorker.register()` is a browser-only API and will throw during SSR if called from the server boundary. (4) **VAPID public key ships as `NEXT_PUBLIC_VAPID_PUBLIC_KEY` env var.** Hardcoding the VAPID key in source creates a git-trackable secret rotation problem. The public key is not sensitive, but the private key is — keep the pair in Railway env vars, expose only the public key via `NEXT_PUBLIC_`. - **Key rotation REMOVED in favor of encrypt-in-place (settled 2026-06-13). SUPERSEDES: the "5-minute window problem" rotation premise, "Key rotation on upgrade", "Every key rotation produces an encrypted key" (2026-04-17), "Pre-rotation chain verification" (2026-04-17), E29 "Restore of rotated keys is blocked outright", E30 "stale-key session-lockout", E31 "block rotate-from-stale", and the "Identity keying is migration-chain-uniform" entry above (the bug class it described is deleted, not fixed).** Validated by three independent agent reviews (irreversibility/extensibility, threat-model, removal-feasibility) on 2026-06-13.

  **New model:** the auto-generated key/address NEVER changes for the life of an identity. No rotation, no on-chain migration, no migration chain, no sweep, no stale-key state. Adding a passphrase ENCRYPTS THE EXISTING key in place (re-encrypt the same WIF under the passphrase → write `bfn_keypair_enc` → remove plaintext). Changing a passphrase decrypts-with-old / re-encrypts-the-same-key-with-new. **Recovery-file export is gated behind having a passphrase** — "Save your account" prompts for a passphrase first, so NO unencrypted recovery file can ever be created or downloaded. This is the load-bearing mitigation: it closes the only realistic plaintext-leak vector (an unencrypted file the user downloads then syncs to the cloud). Restore = import a key from a file (decrypt with passphrase); no eligibility/migration check.

  **Why:** rotation was a correct implementation of an expensive premise ("a plaintext key must be revocable"), but it spawned the entire address-vs-migration-chain bug class (free-boot reset on rotation, blank earnings for a rotated address, creator-bonus paid to a dead address) plus ~2.5–3.5k LOC and the E29/E30/E31 subsystems. Encrypt-in-place is a cheaper premise: keep the plaintext window short, never export an unencrypted file, and accept that a key stolen BEFORE it was ever protected is simply lost. Removing rotation DELETES the bug class rather than fixing it, and a stable key/address makes "15 free boots never reset" and "earnings follow the user" true for free. Pre-launch (fresh-from-post-#1, empty DB) is the only cheap moment — post-launch, once real on-chain migration records exist, removal becomes permanent legacy support.

  **SECURITY REGRESSION ACKNOWLEDGED (Hard Rule #3) — owner signed off 2026-06-13.** This removes in-app REVOCATION of a key that leaked while plaintext (the window between first visit and setting a passphrase — via malware reading browser storage off disk, a device backup, or a shared device). Encrypt-in-place cannot un-leak a copy already taken. Note: E29/E31 (marked FIXED in SECURITY_AUDIT) defended attack surfaces that ROTATION itself created (rotate/restore-from-stale takeover) — those vectors cease to exist once rotation is gone, so they are *obsoleted*, not regressed. The genuinely-lost property is revocation. Accepted at launch scale because: (a) the passphrase-gated-export rule closes the most realistic vector; (b) under the old model the revocation benefit only ever accrued to users who upgraded — most never did and already held an unrevocable plaintext key; (c) re-adding revocation later is cheap and additive. SECURITY_AUDIT.md E29/E30/E31 entries get updated to "superseded by encrypt-in-place" when the code removal lands (Phase 1).

  **Orphaned-data scar is MOOT for OpenCook:** deleting the chain resolver would normally sever a rotated user's old-address earnings — but the fresh-from-post-#1 launch DB has no rotated users, so there is nothing to orphan.

  **Future re-add path (recoverable):** the removal commit will be git-tagged. A future "I think my key was exposed" capability would be an OPT-IN, rare reclaim action — NOT rotation-on-every-upgrade — reusing the git-recoverable migration OP_RETURN format + signature-verification + chain-resolution pieces, not the old rotation-on-protect shape. Keep the on-chain migration payload format documented so a re-add is a design conversation, not archaeology (precedent: the `cleanupMigrations` removal note above).

  **DO NOT** re-introduce rotation-on-passphrase-add, the migration chain resolver, or the E29/E30/E31 machinery without explicitly superseding this decision. **DO** keep the passphrase-gated export rule — no unencrypted recovery file ever leaves the device.

- **Money surfaces key on the BSV address (settled 2026-06-13).** With rotation removed, key == address == pubkey (1:1, stable for life), so every money/attribution lookup, grant, aggregation, and rate-limit bucket standardizes on the BSV ADDRESS as the single canonical identifier. `pubkey` is retained ONLY as a stored attribute for ECDSA signature verification (`createPost`) and as an audit artifact — it is not a lookup key. - **Free-boot path consumes the grant BEFORE paying (settled 2026-06-13, NUANCES C5).** For SERVER-FUNDED free boots, the grant (`free_boots_used`) is consumed before the broadcast, with an idempotency key on the txid/nonce, so a crash between a successful broadcast and the DB write cannot let the user retry and make the server pay twice. - **Paid-boot confirm records from on-chain outputs, never recompute-and-reject (settled 2026-06-15, deep-audit Finding 6).** `/api/boot-confirm` used to RE-derive the contribution split (price × weights) and reject (HTTP 400) if the already-broadcast tx's outputs didn't match. But weights/price legitimately drift between the boot-shares quote and the confirm POST (every other boot mutates `bootboard` → weights; a new post moves the price), so this rejected honest, already-on-chain txs → the client retried → built a NEW tx (new txid) → DOUBLE-PAID (the replay guard is txid-only). New model: the tx is self-authenticating (`hash(rawTx)===txid`) and already broadcast, so boot-confirm RECORDS the payouts FROM the parsed on-chain outputs (`txVouts`) — it does not re-adjudicate the split. **One enforced conservation invariant:** the platform must receive ≥ `floor(bootPriceFloor × platformCut) − 2` sats on-chain (uses the FLOOR, not the drift-prone recomputed price, so it never false-rejects; the platform address is the server wallet, never the booter). **Client (`useBoot`):** after a successful broadcast it NEVER rebuilds a new tx on confirm failure — it re-POSTs the SAME confirm body (idempotent: a duplicate hits the 409 txid replay guard) on transient 503/network-throw, and only rebuilds on TX_CONFLICT (the tx never landed). **Accepted residual (Hard Rule #3 — net STRONGER than before, which had no total floor at all):** a client can under-pay the *non-platform* contributors and still get a boot recorded + bootboard credit — but that's their own money, contributors are credited exactly what was sent on-chain (honest ledger), and a per-contributor floor is impossible because the booter can themselves be a pool recipient (it would false-reject a solo contributor). `recipient_pubkey` is now just an audit label (earnings joins on `recipient_address` — see "Money surfaces key on the BSV address"). The booter's own change output (and, as a minor consequence, a booter-contributor's own pool share) is excluded from recorded payouts. Code-auditor verified the double-pay vector is closed (incl. the thrown-fetch path) and no forged/phantom payout is possible. The FREE/server-funded path (`boot-orchestrator`) is unchanged — it correctly trusts its own server-built split because the server IS the payer there.

- **Balance shows spendable (confirmed); deposit shortfall includes the network fee (settled 2026-06-15, device-test findings).** On-device QA showed the balance pill counting unconfirmed (0-conf) UTXOs as spendable — displaying ~7,687 when only 5,023 was confirmed/spendable — and a paid boot failing "not enough funds" because the displayed cost omitted the network fee (5,023 − 4,836 = 187 < fee). Both are display/affordability-honesty bugs, NOT money-loss (every sat was always accounted on-chain). Decisions: (1) `/api/balance` returns `balance` = CONFIRMED (spendable) only, plus a separate `pending` (0-conf, by WoC `height`). The UI headline is spendable; pending shows as a muted "+X pending" line. Rationale: the displayed number must answer "what can I spend right now" — a pending-inclusive total made boots look affordable when they weren't, and made the balance look like it didn't drop after a boot (pending change re-inflated it). (2) The deposit/top-up shortfall is measured against price + network fee, not price alone. `clientSideBoot` surfaces the fee it already computes — `estimateFee(1, outputCount)`, the exact threshold the `insufficient_funds` branch tests against — through `useBoot` → `Feed` → `FundAddress`. Because that branch is only reached when `balance < price + fee` (the `needs_consolidation` branch creams off the affordable-after-consolidation case first), the shortfall is provably always strictly positive, so the modal can never again say "you have enough" after a real failure. Auditor-verified PASS: pure value-surfacing, NO change to coin selection / fee estimation / signing / broadcast / spent-set / 0-conf chaining. (3) **SEMANTICS NOTE — do not conflate two `balance` numbers:** `FundAddress.balance` (and the displayed pill) now means CONFIRMED/spendable, while `clientSideBoot`'s internal `balance` (~line 381) still sums ALL fetched UTXOs because it legitimately spends 0-conf change via 0-conf chaining. They mean different things BY DESIGN. No security control affected. **DO NOT** revert `/api/balance` to a confirmed+unconfirmed sum, or drop the fee from the affordability/shortfall math.

- **Broadcast timeout is INDETERMINATE — never rebuild (settled 2026-06-16, Phase 2 Build A).** The server wallet (`wallet.ts`, funds free boots + post-logging) holds a mutex across 4 external calls that had NO timeouts — any hang froze ALL free boots + post-logging site-wide until the socket died. Build A added timeouts: **10s on the read calls** (unspent / source-tx-hex / double-spend lookup — abort + fail-safe, no money moved, safe to retry) and **30s on `tx.broadcast()`**. The load-bearing rule: a broadcast timeout does NOT mean the tx failed — ARC may have accepted it slowly. So it returns a distinct terminal `broadcast_timeout` status that **NEVER triggers a rebuild** (a rebuilt tx = a new txid = the server DOUBLE-PAYS). On timeout the wallet **releases the UTXO reservation but does NOT blacklist the inputs** (WhatsOnChain reflects on-chain reality; the change output resurfaces via WoC if the tx landed; rare stale-WoC reuse is caught by the existing `DOUBLE_SPEND_ATTEMPTED` self-heal) and does NOT register change. Both retry sites skip it: `boot-payment.ts`'s 1s auto-retry and `onchain.ts`'s post-log retry only re-run the did-not-broadcast failures. The free grant was already consumed pre-broadcast and is NOT refunded (consistent with "Free-boot path consumes the grant BEFORE paying"); the orchestrator surfaces `indeterminate:true` → `bootPost` returns it with **NO `error`** → `useBoot` releases quietly with **NO "tap to retry"** (a manual retry would rebuild a fresh tx and double-pay the same post). Auditor-verified PASS: the mutex always releases (outer `finally`), no `broadcast_timeout` path can reach a rebuild, `withTimeout` clears its timer + a late-resolving broadcast can't mutate state, no security control weakened. **DO NOT** make a broadcast timeout retry/rebuild, and **DO NOT** blacklist-on-timeout (a false timeout would strand the wallet's main UTXO until restart). Read-call timeouts ARE safe to retry. Builds B (balance precheck → route free→paid + low-balance alert), C (env-var kill-switch, fail-closed, pre-consume), D (broadcast proxy) follow.

- **Dry server wallet routes free boots to paid (settled 2026-06-16, Phase 2 Build B).** Previously a dry server wallet consumed the user's free-boot grant THEN failed at broadcast with no refund — burning the grant AND erroring (the device-test rough edge). Build B adds a balance precheck in `executeBoot` BEFORE the grant consume: read `getBalance()` (spendable); if it can't cover `actualPrice + SERVER_FEE_BUFFER_SATS`, route the boot to PAID (`isFree:false`, real dynamic `getBootPrice(db)` → `bootPost` returns `requiresPayment`), exactly like the `FREE_GRANT_EXHAUSTED` path. The fee figure is EXACT: a split's outputs total === `actualPrice` (no rounding drift — the `Math.floor` remainder folds into the creator output), and `SERVER_FEE_BUFFER_SATS` (500) is now exported from `wallet.ts` and used by BOTH the precheck and `reserveUtxos`'s `totalNeeded` — so the precheck can NEVER green-light a boot that then hits `insufficient_funds` and burns the grant. **Fails toward PAID:** a WoC read failure makes `getBalance` read low/0 → paid, never fail-open into a doomed broadcast (the precheck cannot throw — `getUtxos` swallows all errors to the `_pendingChange` fallback). The precheck sits BEFORE the consume, so it does NOT relitigate "no refund on broadcast failure" — it avoids REACHING the consume when the server can't pay. Plus a debounced (5-min) low-balance console alert at `serverLowBalanceAlertSats` (10k sats, `config.ts`) — alert-only, never blocks, fires on the free-boot path (where the server drain is). Auditor PASS (no grant-burn path, no fail-open, fee-buffer sync exact). Accepted residuals: one extra WoC fetch per free boot (Build D cache-reuse), a NARROW precheck→reserve TOCTOU that can still burn one grant under concurrent UTXO contention (Build B reduces, doesn't eliminate, the burn window — consistent with the settled no-refund policy). Real alerting (webhook/email) is Phase 5. **DO NOT** move the precheck after the grant consume, and keep the precheck threshold in sync with `SERVER_FEE_BUFFER_SATS`.

- **Server-wallet kill-switch — env var, fail-closed, pre-consume (settled 2026-06-16, Phase 2 Build C).** `isServerSpendDisabled()` (`wallet.ts`) reads `BSV_WALLET_SPEND_DISABLED` (truthy `"true"`/`"1"` = tripped). When tripped the server wallet STOPS spending: (1) `executeBoot` checks it at the TOP of the `if (isFree)` block — BEFORE the grant consume — and routes free boots to PAID (`isFree:false`, `error:"SERVER_SPEND_DISABLED"` → `bootPost` → `requiresPayment`), so tripping it consumes NO grant; (2) `buildAndBroadcast` returns a new terminal `{status:"spend_disabled"}` as a backstop that also stops post-logging + any direct server spend (skipped in both retry sites). **Paid/client boots are UNAFFECTED** — they spend the user's own funds via `client-boot.ts`, and boot-confirm's server re-broadcast uses `parsed.broadcast()` (NOT the gated `buildAndBroadcast`), so killing free boots can't break paid boots (auditor-confirmed the one over-reach risk). Env var (v1) → takes effect on redeploy; a DB-backed instant runtime toggle is the documented fast-follow (the auditor's "fail-closed on unreadable" applies to that DB read, not to an env read which can't throw — `process.env.X ?? ""`). Default unset = spending enabled (correct for an env var — you don't want spending dark by default). Only exact `true`/`1` trip it (a typo reads as enabled = the safe direction). Unit-tested (tripped → no grant consumed, no broadcast, routes to paid). Auditor PASS (a–f). **DO NOT** gate the paid/client path or move the switch check after the grant consume.

- **Durable post-retry: a post broadcast TIMEOUT re-sweeps — distinct from the boot no-rebuild rule (settled 2026-06-19, Phase 4 Wave 1).** Locked invariant: *every accepted post must land on-chain — no off-chain orphans.* Before this, a post whose OP_RETURN broadcast timed out (or failed after the inline 1s retry) returned `null` and was **orphaned forever** (`tx_id` stayed NULL, nothing re-attempted it). Wave 1 makes `tx_id IS NULL` a durable retry queue drained by an ambient-traffic single-flight sweep (`src/services/bsv/anchor-sweep.ts` `sweepOrphans()`, fired fire-and-forget from `createPost` and `GET /api/posts`; in-memory exponential backoff; one broadcast per sweep; 90s min-age so it never races the inline first attempt). **This intentionally DIVERGES, for POSTS ONLY, from "Broadcast timeout is INDETERMINATE — never rebuild" (the Phase 2 Build A entry above):** that rule exists because a rebuilt BOOT tx mints a new txid and the server **double-pays contributors** — a *financial* hazard. A post-log has **NO payee** (a 0-sat OP_RETURN), so re-broadcasting a timed-out post costs — in the rare case the original actually landed — **one wasted ~66-sat OP_RETURN** (~$0.0000077, a rounding error), not a double-pay. NOT retrying would instead permanently orphan a never-landed timeout, violating the invariant. So **boots keep the strict no-rebuild rule; posts re-sweep until anchored** — the two policies coexist by design, keyed on the no-payee property, NOT a drift. A rare duplicate on-chain record is harmless (the `onchain-record` reader contract keys on `(app,type)`; the fairness engine attributes from the one DB row per post, never from counting OP_RETURNs). Zero schema change (the queue is `tx_id IS NULL`); in-memory backoff lost-on-restart = an immediate retry after deploy, which is desirable. Kill-switch / dry-wallet stragglers stay NULL and drain gradually (one-at-a-time on ambient traffic, not a flood) once spending resumes. **DO NOT** extend re-broadcast-on-timeout to the boot path (`boot-payment.ts` / `boot-orchestrator.ts`) — that reopens the double-pay vector Phase 2 Build A closed.

- **Phase 4 abuse/cost caps — all-posts-on-chain + refuse-gates + spend ceiling + boot-price anti-inflation (settled + built 2026-06-19, Waves 0–3).** Locked principle: **every ACCEPTED post must land on-chain — no off-chain orphans.** A post the server can't/won't fund on-chain is REFUSED upfront, never stored. Built lean — the heavy options (a setInterval retry worker, a DB-backed budget) were deliberately rejected as over-engineering per the 2026-06-13 "neutral extensibility headroom" decision.
  - **Wave 0 — rate-limiter correctness fix** (`rate-limit.ts`): cleanup pruned every key against the FIRST caller's window; a 60s-window caller registering first pruned the 24h free-boot-cap's timestamps after ~60s, silently defeating that cap (server-wallet drain). Now each entry stores its own window. A real latent bug in a SECURITY_AUDIT-FIXED control.
  - **Wave 1 — durable on-chain retry** (`anchor-sweep.ts`): see "Durable post-retry: timeout => re-sweep" above. Money-path auditor CLEAN.
  - **Wave 2 — refuse-gates + daily spend ceiling.** `createPost` runs three gates BEFORE the insert (so a refused post is never stored off-chain): (1) per-IP **200/day** on-chain-post cap (env `ONCHAIN_POST_IP_LIMIT`, reuses `rateLimit`) → refuse `daily_limit`; (2) `isServerSpendDisabled()` → refuse `paused` (kill-switch now REFUSES new posts rather than insert-then-strand — closes a prior orphan source); (3) `hasDailyBudget()` → refuse `paused`. The **in-memory daily spend ceiling** (`server-spend-budget.ts`, env `SERVER_DAILY_SPEND_SATS`, ~$0.20/day default) caps total server-wallet spend across BOTH post-logging AND free boosts (shared wallet) — the aggregate backstop behind the per-IP caps. `bootPost` checks the ceiling BEFORE consuming the grant/IP-cap → routes free→paid when blown (boost still happens, user funds it; no grant burned). BLOCK-at-200 protects all three surfaces (wallet, boot-price, DB), vs the rejected skip-logging which protected only the wallet. **In-memory by design:** a redeploy could let one UTC day exceed the ceiling once (~2x ~$0.20 = ~$0.40, trivial vs the ~$50/mo budget). **Accepted residual (OBS-2):** `hasDailyBudget`/`recordDailySpend` aren't atomic — N concurrent requests at the boundary over-admit by N×cost (cents); bounded, consistent with the ~2x tolerance, not worth refund complexity. **Indeterminate free-boost broadcasts ARE recorded** (OBS-1 fix — presume-spent, the safe direction for a ceiling).
  - **Wave 3 — boot-price anti-inflation** (`pricing.ts`): the dynamic price counted `COUNT(DISTINCT pubkey)`, so a distributed attacker minting one fresh identity per post could pin the price at the 250k ceiling, griefing real payers. Now only ESTABLISHED contributors count — pubkeys with ≥ `minPostsForPricing` (3) posts in the window (`HAVING COUNT(*) >= 3`). **Pricing query ONLY** — the payout split (`weights.ts`/`split.ts`) does NOT need the gate (a 1-post identity earns a vanishing `sqrt(1)×decay` share, not free money); gating payouts would be a separate economic-policy choice, deliberately NOT bundled.
  - **DEFERRED (no invariant weakened):** rate-limiter LRU key-cap (store self-cleans), persisting the agent daily counter (in-memory, matches existing design), prompt `cache_control` (needs a stable-prefix refactor first; post-launch cost pass). The user-funded posting escape valve stays deferred — at a 200/day block it only bites abuse, but it's the eventual companion so a blocked user always has a paid route on-chain.
  - **Forward-compat (future idea-attribution):** a rare duplicate on-chain post record (Wave 1 timeout re-sweep) is SAFE for any future "pay the idea for its contribution" mechanism — it must key on EARLIEST on-chain timestamp + dedupe by idea (content/signature), NOT naively sum on-chain occurrences. Same author (pubkey/sig), later timestamp → the original is canonical, the duplicate ignored. ("First to say it on-chain" wins.)
  - **Deploy (OBS-3/OBS-4):** per-IP caps require a trusted proxy setting `x-forwarded-for`/`x-real-ip`; without one, header-less requests share a `postIp:unknown` bucket (fails toward REFUSED — an availability cliff, not a money hole). New env vars `SERVER_DAILY_SPEND_SATS` + `ONCHAIN_POST_IP_LIMIT` documented in `.env.example`. **DO NOT** make the refuse-gates run after the insert (would store off-chain), and **DO NOT** add the ≥3 gate to the payout split without an explicit economic-policy decision.
